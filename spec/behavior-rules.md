# 行为规范

版本: v0.1

## 1. 先 Spec，后实现

每个功能进入实现前，必须先完成 Spec 探讨并获得确认。

Spec 至少包含：

- 目标
- 输入与输出
- 用户流程
- 边界情况
- 验收标准
- 测试用例

未确认 Spec 前，不做代码实现，不改项目结构，不引入新依赖。

## 2. 所有产出必须沉淀测试

每新增一个能力，都必须同步增加或更新测试用例。

测试用例包括但不限于：

- 单元测试
- 集成测试
- RAG 检索测试
- 问答评测样例
- 文档解析回归样例

对于 RAG 问答能力，测试样例至少应包含：

- 问题
- 期望答案要点
- 期望引用来源
- 允许的答案变体
- 不应出现的幻觉内容

## 3. 参考 RAG-cy，但不照搬

`RAG-cy/` 是参考项目，主要借鉴：

- PDF 解析流程
- chunk 策略
- 父文档检索
- rerank 思路
- 结构化回答
- 来源引用

不直接照搬：

- 竞赛型临时代码
- 缺少测试的实现方式
- 强绑定特定比赛数据结构的逻辑
- 过度复杂且当前 MVP 用不到的配置

## 4. 实验室知识库优先保证可信

知识库回答必须优先基于指定文档：

- `lab-rag-qa/data/examples/江涛组相关实验SOP及附录---2020.5.19 (2).pdf`
- `lab-rag-qa/data/examples/gao lab protocol.pdf`

回答要求：

- 能引用文档时，必须给出来源。
- 文档未覆盖时，明确说明无法从当前知识库确认。
- 不把模型常识伪装成文档事实。
- 对实验安全、剂量、时间、温度、浓度等关键信息保持保守。

## 5. 每轮工作闭环

每轮工作应尽量形成闭环：

- 明确本轮目标。
- 更新或确认 Spec。
- 实现前说明将修改的范围。
- 实现后运行相关测试。
- 交付时说明改动、测试结果和剩余风险。
